# ReproRun

> 論文を渡せば、その結果が本当に再現できるかどうかを教えてくれます。

**ReproRun** は、互換性のあるあらゆる AI コーディングアシスタントで利用できる、ポータブルな **AI エージェントスキル**です。「ある論文が X を主張している」から「X が実際に自分のマシンで動く」までの、つらい道のりを自動化します。論文は美しい数値を提示しますが、その再現はたいてい壊れたコード、ビルドできない環境、バージョンのずれによって頓挫します。ReproRun はパイプライン全体をエンドツーエンドで処理します。

**論文を読む → コードとデータを見つける → 環境を構築する → スモークテスト → フル実行 → 測定された数値を論文の主張と比較する。**

---

## ✨ あらゆるプラットフォームに適応するよう設計

ReproRun は、ひとつの固定されたセットアップを前提とするのではなく、**渡されたものに自ら適応する**よう設計されています。

- **クロス OS** — Windows、macOS、Linux で動作
- **クロス言語** — Python では完全なパイプライン、R / MATLAB / Julia では最小限の経路
- **クロスドメイン** — シングルセル生物学、画像 ML など、ドメインごとの再配線なし
- **クロスモード** — *tool mode*（`pip install` + 呼び出しスクリプトの作成）または *experiment mode*（リポジトリをクローンしてそのスクリプトを実行）— 論文ごとに自動選択

---

## 🚀 何ができるか

- **タイトルだけから論文を見つける** — PDF とコードリポジトリを自動でウェブ検索
- **環境の劣化を自動診断** — numpy の ABI 衝突、torchvision の API 変更、非推奨になった pandas メソッドなどを自動で検出して修正
- **パラメータを推測しない** — インストール後に関数のシグネチャを調べる
- **5 ラウンドの依存関係セルフヒーリングループ** — エラーを分類 → 的を絞った修正 → 再検証、最大 5 ラウンド
- **論文の分離** — それぞれの論文が独自の出力名前空間を持つ

---

## 🏗️ アーキテクチャ

ひとつのオーケストレーター（`SKILL.md`）が 6 つの専門エージェントを駆動します。

| エージェント | 役割 |
|-------|------|
| A · paper-reader | 再現すべき数値的な主張を抽出する |
| B · resource-finder | コードリポジトリとデータセットを特定する |
| C · environment-builder | 実行環境を構築・修復する（最も複雑） |
| D · smoke-tester | 簡易スモークテスト — 動作を確認する |
| E · full-runner | フル再現実行 |
| F · result-comparator | 測定値と主張値を項目ごとに比較する |

## ✅ 検証

ReproRun は、ドメインをまたぐ実際の論文でエンドツーエンドに実行されてきました。単に「成功」というスタンプを押すだけではありません。各論文について、**正直な評価**を返します。数値が再現された、パイプラインが再現された、または *現状のままでは再現できない* — 常に根本原因とともに示します。

| 論文 | ドメイン | 結果 |
|-------|--------|--------|
| **UMAP** (McInnes 2018, JOSS) | 次元削減 | ✅ **数値が再現された** — 14 個中 11 個の k-NN 精度指標が ±0.01 以内で一致。MNIST と Fashion-MNIST は小数 3 桁まで確認 |
| **scVelo** (Bergen 2020, Nat Biotech) | シングルセル | ✅ **パイプラインが再現された** — 100% NaN を引き起こす numpy 2.x の ABI バグを検出し、1.26.4 へのダウングレードで修正 |
| **Robust Stitching** (Ruiz 2023, ICML) | 画像 ML | ✅ **パイプラインが再現された** — 5 件の劣化による不具合を修復（torchvision API、pandas `append`、不足していた依存関係） |
| **Annotatability** (Nitzan 2024, Nat Comp Sci) | シングルセル | ✅ **パイプラインが再現された** — 6 ラウンドの API デバッグで `pooch` 依存関係の不足が判明 |
| **ScType** (Ianevski 2022, Nat Comms) | シングルセル (R) | ✅ **パイプラインが再現された** — バージョンのダウングレード後に R 経路を検証 |
| **Cropformer** (Wang 2025, Plant Communications) | 作物ゲノミクス | ⚠️ **部分的** — コード、モデル、学習は検証済みだが、リポジトリには 10 件のデモサンプルしか含まれないため、論文の PCC=0.92 は現状のままでは再現できない |

**論文 6 本 · 全指標の再現 1 件 · パイプライン再現 4 件 · 正直な部分再現 1 件 · 検証済みスキル改善 18 件**

> **設計上、正直であること。** UMAP は指標一致率 78.6% に着地しました。これは私たちの 80% の「クリーンな再現」基準を*わずかに下回る*もので、ReproRun はこれを切り上げるのではなくデータの矛盾として報告します。Cropformer については、フレームワークはエンドツーエンドで動作しますが、公開された数値にはリポジトリが決して提供しない本物の作物ゲノムデータが必要です。そのため Pass ではなく Partial としてフラグが立てられます。

<details>
<summary><b>ケーススタディ — Cropformer（⚠️ 部分的な再現）</b></summary>

**検証済み ✅**
- リポジトリを発見してクローン（`jiekesen/Cropformer`。論文の URL は誤っていた）
- 環境を構築 — Python 3.10 + PyTorch 2.5.1 + CUDA 12.1
- モデルアーキテクチャ — Conv1d + 8 ヘッドの self-attention（パラメータ 2.6M）
- RTX 4090 での GPU 推論。学習済みの重みが読み込まれ動作する
- 学習ループが収束 — loss 89,540 → 23,771

**再現できなかった ❌**
- 論文の指標（PCC=0.92 など）— リポジトリには本物の作物データではなく、ランダムなデモサンプル 10 件しか含まれない
- 分類タスク — `model_class.py` に主要な関数が欠けている
- ネストされた交差検証、MIC 特徴選択、0〜9 の SNP エンコーディング — リポジトリに実装されていない

**根本原因:** 公開リポジトリはデモ専用であり、完全なパイプライン（MIC 選択、ネストされた CV、Optuna チューニング）と本物のデータセットは含まれていません。忠実な再現には、本物の作物ゲノムデータ → PLINK 処理 → 記述されたパイプラインの再実装（データと計算作業で約 1〜2 週間）が必要になります。
</details>

## 📦 はじめに

ReproRun は、互換性のあるあらゆる AI コーディングアシスタントで動作するエージェントスキルです。使うには：

1. スキルを読み込める AI コーディングエージェントを使う。
2. `paper-reproduction/` フォルダを、エージェントがスキルを読み込む場所に置く。
3. セッション中に、ただ頼むだけ — 例：*"reproduce scVelo"* や *"reproduce Table 2 from this paper."* スキルが自動的に起動します。
